Successful Purchase & Activation
The most important flow — customer buys Pro, receives their licence key by email, and activates it in the desktop app. Click any step to expand the technical detail behind it.
● Customer / Browser ● Cloudflare Worker ● Stripe ● D1 Database ● Resend Email
Step-by-Step Flow
All 23 steps from "Buy Now" click to Pro features unlocked. Click any step for technical detail.
Phase 1 — Customer Initiates Checkout
1
● Customer
Customer clicks "Buy Now" on modelcollectr.co.uk
The pricing page or nav CTA triggers a JavaScript function — no redirect to Stripe yet.
The button calls startCheckout(), a JS function in the page that makes a fetch request to the Worker API. The customer never gets a static Stripe Payment Link — everything goes through the Worker so the success page flow works correctly. 
// modelcollectr.co.uk/index.html
async function startCheckout() {
  const res = await fetch('https://api.modelcollectr.co.uk/checkout', {
    method: 'POST'
  });
  const { url } = await res.json();
  window.location.href = url; // redirect to Stripe
}
2
● Worker
Worker receives POST /checkout request
CORS origin is validated — only modelcollectr.co.uk domains and localhost are permitted.
The Worker checks the Origin header against an allowlist:
  • https://modelcollectr.co.uk
  • https://www.modelcollectr.co.uk
  • https://test.modelcollectr.co.uk
  • http://localhost:8767
Any other origin gets a CORS error and no Stripe session is created.
3
● Worker
Worker calls Stripe API to create a Checkout Session
A hosted payment page is created with the annual price, 14-day free trial, and promo codes enabled.
POST https://api.stripe.com/v1/checkout/sessions

mode:                 subscription
line_items[0][price]: price_1TXsXDE8AmKiRjJbNQU0xGdu  (£19.99/yr)
line_items[0][quantity]: 1
success_url:          https://modelcollectr.co.uk/success?session_id={CHECKOUT_SESSION_ID}
cancel_url:           https://modelcollectr.co.uk/#pricing
allow_promotion_codes: true
subscription_data[trial_period_days]: 14
The {CHECKOUT_SESSION_ID} placeholder is filled in by Stripe automatically.
4
● Stripe
Stripe creates the session and returns a hosted checkout URL
Stripe responds with a unique URL to its hosted payment page. This URL expires after 24 hours.
Stripe returns a session object. The most important fields: 
{
  "id": "cs_test_...",
  "url": "https://checkout.stripe.com/c/pay/cs_test_...",
  "payment_status": "unpaid",
  "status": "open"
}
The Worker extracts session.url and returns it to the browser as {"url": "https://checkout.stripe.com/..."}.
5
● Customer
Browser redirects customer to Stripe's hosted checkout page
Customer sees the Stripe payment form with ModelCollectr branding, the annual plan details, and a 14-day free trial.
The customer is now on checkout.stripe.com — fully handled by Stripe. Your server is not involved here. The customer sees:
  • Product: ModelCollectr Pro — £19.99/year
  • 14-day free trial (no charge today)
  • Promotion code box (if they have a code)
  • Email, card number, expiry, CVC fields
Stripe handles 3D Secure authentication automatically if the card requires it.
6
● Customer
Customer enters card details and clicks Subscribe
Stripe validates the card, handles 3D Secure if needed, and creates the subscription. The customer is not charged today (14-day trial).
Stripe collects and tokenises the card details server-side — the card number never touches your server. When Subscribe is clicked:
  • Stripe creates a Customer object (with the email they entered)
  • Stripe creates a Subscription object (status: trialing)
  • Stripe creates an Invoice for £0.00 (trial period)
For testing, use card 4242 4242 4242 4242 with any future date and any CVC.
Phase 2 — Stripe Fires Webhook → Worker Processes
7
● Stripe
Stripe fires customer.subscription.created webhook
As soon as the subscription is created, Stripe sends an HTTP POST to the Worker's webhook endpoint. This happens in parallel with redirecting the customer.
Stripe sends to https://api.modelcollectr.co.uk/webhooks/stripe: 
POST /webhooks/stripe
Stripe-Signature: t=1716000000,v1=abc123...,v0=...

{
  "type": "customer.subscription.created",
  "data": {
    "object": {
      "id": "sub_1ABC...",
      "customer": "cus_1XYZ...",
      "status": "trialing",
      "current_period_end": 1748649600
    }
  }
}
The webhook may arrive before or after the customer lands on the success page.
8
● Worker
Worker verifies the webhook signature (HMAC-SHA256)
Every webhook is verified using a secret shared with Stripe. Webhooks older than 5 minutes are rejected to prevent replay attacks.
The Stripe-Signature header contains a timestamp and HMAC-SHA256 hash. The Worker:
  • Extracts the timestamp t= from the header
  • Rejects if the timestamp is more than 300 seconds old
  • Computes HMAC-SHA256(timestamp.rawBody, STRIPE_WEBHOOK_SECRET)
  • Compares using constant-time string comparison (prevents timing attacks)
If verification fails, the Worker returns HTTP 400. Stripe will retry.
9
● Worker
Worker checks idempotency — has this subscription already been processed?
Stripe can deliver the same webhook more than once. The Worker checks D1 to avoid issuing a second key for the same subscription.
SELECT key FROM licenses
WHERE stripe_subscription_id = 'sub_1ABC...'
If a row already exists → the Worker breaks out of the event handler immediately and returns HTTP 200 (telling Stripe not to retry). This means even if Stripe retries 3 times, the customer only ever gets one key.
10
● Stripe
Worker fetches the customer's email address from Stripe
The webhook contains the customer ID but not the email. The Worker makes a quick API call to get it. Email is never stored — only used to send the email, then discarded.
GET https://api.stripe.com/v1/customers/cus_1XYZ...

Response:
{
  "id": "cus_1XYZ...",
  "email": "customer@example.com",
  "name": "Jane Smith"
}
The Worker uses customer.email only to (a) send the licence email and (b) that's it. The email is never written to D1. This is the GDPR-safe design — Stripe holds the personal data, we hold only the anonymous licence key and Stripe IDs.
11
● Worker
Worker generates a cryptographically secure licence key
A 16-character key in MC-XXXX-XXXX-XXXX-XXXX format is generated using the browser's Crypto API — not Math.random().
function generateKey() {
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789';
  // 32-char alphabet: excludes 0/O and 1/I to avoid confusion
  const bytes = crypto.getRandomValues(new Uint8Array(16));
  let i = 0;
  const group = () => Array.from(
    { length: 4 },
    () => chars[bytes[i++] % chars.length]
  ).join('');
  return `MC-${group()}-${group()}-${group()}-${group()}`;
}
// → "MC-K8HN-4XRQ-BZTW-7YJP"
crypto.getRandomValues() uses the OS's cryptographic random source — it is not predictable. The old Math.random() was replaced as a security fix.
12
● D1 Database
Worker writes the new licence to D1
A row is inserted with the key, Stripe IDs, status 'active', and a timestamp. No email address is stored — it stays in Stripe.
INSERT INTO licenses
  (key, stripe_customer_id, stripe_subscription_id, status, created_at)
VALUES
  ('MC-K8HN-4XRQ-BZTW-7YJP',
   'cus_1XYZ...',
   'sub_1ABC...',
   'active',
   1716000000000)
D1 schema: key (PK), stripe_customer_id, stripe_subscription_id, status, created_at, last_check_in, check_in_count, notes. All are non-personal. The email address lives only in Stripe.
13
● Resend Email
Worker sends the licence key email via Resend
A branded HTML email is delivered to the customer's address with their key, step-by-step activation instructions, and support contact.
POST https://api.resend.com/emails
Authorization: Bearer re_hT9...

{
  "from": "ModelCollectr <hello@modelcollectr.co.uk>",
  "to": ["customer@example.com"],
  "subject": "Your ModelCollectr Pro Licence Key",
  "html": "..."
}
The email includes the key in a large monospace font box, and 4-step activation instructions (Help → Enter Licence Key → paste → Activate). Resend delivers from the verified domain modelcollectr.co.uk.
Phase 3 — Success Page Polling
14
● Stripe
Stripe redirects customer to the success page
After payment, Stripe redirects to success.html with the session ID in the URL. This happens independently of the webhook.
Stripe redirects to: 
https://modelcollectr.co.uk/success?session_id=cs_test_abc123...
The session ID allows the success page to identify which purchase just happened and retrieve the correct licence key from the Worker.
15
● Customer
Success page loads and begins polling the Worker
The page shows a loading spinner and polls every 2 seconds, waiting for the webhook to be processed and the key to appear in D1.
// success.html polls this endpoint:
GET /license/lookup?session_id=cs_test_abc123...

// Success page polls every 2 seconds, up to 30 attempts (1 minute):
async function poll() {
  const res = await fetch(`/license/lookup?session_id=${sessionId}`);
  const data = await res.json();
  if (data.ready) showKey(data.key, data.email);
  else setTimeout(poll, 2000);
}
The webhook and the redirect happen in parallel. Usually the key is in D1 by the time the success page makes its first request.
16
● Stripe
Worker fetches the Checkout Session from Stripe to get the customer ID
The session ID from the URL is used to look up the Stripe session and retrieve the stripe_customer_id and the email (from customer_details).
GET https://api.stripe.com/v1/checkout/sessions/cs_test_abc123...

Response:
{
  "customer": "cus_1XYZ...",
  "customer_details": { "email": "customer@example.com" },
  "payment_status": "paid"
}
The email comes from session.customer_details.email — no extra Stripe API call needed. If customer is null, the webhook hasn't been processed yet → return {ready: false}.
17
● D1 Database
Worker looks up the licence key by customer ID
D1 is queried for an active licence matching this Stripe customer. If found, the key is returned. If not (webhook not yet processed), returns {ready: false}.
SELECT key FROM licenses
WHERE stripe_customer_id = 'cus_1XYZ...'
  AND status = 'active'

→ { key: 'MC-K8HN-4XRQ-BZTW-7YJP' }
If no row is found yet, the Worker returns HTTP 202 with {"ready": false}. The success page waits 2 seconds and tries again.
18
● Customer
Success page displays the licence key and confirmation message
The spinner is replaced by the licence key in a large box with a "Copy" button, plus a note that the key has been emailed to their address.
The Worker returns {"ready": true, "key": "MC-K8HN-4XRQ-BZTW-7YJP", "email": "customer@example.com"} and the success page:
  • Shows the key in a large monospace display
  • Shows "We've emailed a copy to customer@example.com"
  • Provides a one-click copy button
  • Shows 4-step activation instructions
Phase 4 — App Activation
19
● Customer
Customer opens ModelCollectr and navigates to Enter Licence Key
In the app: Help menu → Enter Licence Key. A dialog appears with a text field. The customer pastes or types the key.
The app should accept the key in any case (upper or lower — it normalises to uppercase before sending). Keys in the format MC-XXXX-XXXX-XXXX-XXXX where X is A-Z, 2-9 (excluding 0, O, 1, I to avoid ambiguity).
20
● Worker
App sends POST /license/verify to the Worker
The Worker validates the key format with a regex, looks it up in D1, updates the check-in stats, and returns the validity result.
POST https://api.modelcollectr.co.uk/license/verify
Content-Type: application/json

{"key": "MC-K8HN-4XRQ-BZTW-7YJP"}

// Worker validates format first:
/^MC-[A-Z2-9]{4}-[A-Z2-9]{4}-[A-Z2-9]{4}-[A-Z2-9]{4}$/
Invalid format → returns {"valid": false, "reason": "invalid_format"} immediately without touching D1.
21
● D1 Database
Worker looks up the key and updates check-in stats
Two queries: a SELECT to get the key's status, then an UPDATE to record the check-in timestamp and increment the counter.
SELECT key, status FROM licenses
WHERE key = 'MC-K8HN-4XRQ-BZTW-7YJP'

UPDATE licenses
SET last_check_in = 1716000000000,
    check_in_count = check_in_count + 1
WHERE key = 'MC-K8HN-4XRQ-BZTW-7YJP'
The check-in count and last check-in date are visible in the admin dashboard — useful to see if a customer is actively using the software.
22
● Worker → Customer
Worker returns {valid: true, status: 'active'}
The response confirms the key is valid and active. The app receives this and unlocks all Pro features immediately.
Response:
{
  "valid": true,
  "status": "active"
}
The app should cache this response locally (with a timestamp) to support the offline grace period. On every startup, the app checks in with the Worker — if it can't reach the API, it falls back to the cached result for up to 7 days.
23
● Customer
Pro features are unlocked in ModelCollectr
The customer has full Pro access. The app will check in with the Worker on each startup to confirm the subscription remains active.
The app should store the licence key and last-valid timestamp persistently (e.g. Windows Registry or local config file). The key is the only credential needed — all status is managed server-side via Stripe and D1.
