Customer Cancellation
Customer decides to cancel their subscription. Stripe lets them keep access until the end of the billing period they've already paid for, then the key is revoked automatically. No refund is involved — they paid for the year.
● Customer ● Stripe ● Cloudflare Worker ● D1 Database
Step-by-Step Flow
6 steps. The customer retains access until their paid period ends. Click any step for detail.
1
● Customer
Customer requests cancellation
Customer either contacts you directly (email support) or uses the Stripe customer portal link to cancel themselves.
Two ways this can happen:
  • Self-serve: Customer visits the Stripe customer portal (you can give them the URL from Stripe Dashboard → Customer → "Send customer portal link"). They cancel there directly.
  • Via you: Customer emails asking to cancel. You go to Stripe Dashboard → Subscriptions → find their subscription → Cancel.
In both cases, Stripe gives the option to cancel "immediately" or "at period end". Always choose at period end unless it's a refund situation.
2
● Stripe
Stripe marks subscription cancel_at_period_end = true
The subscription doesn't end immediately. Stripe sets a flag that means "cancel when the current period expires". The customer keeps their access.
At this point, the subscription object has: 
{
  "status": "active",
  "cancel_at_period_end": true,
  "cancel_at": 1748649600,
  "current_period_end": 1748649600
}
A customer.subscription.updated webhook fires to tell the Worker about this change, but since status is still active, the Worker keeps the licence active in D1.
3
● Customer
Customer retains Pro access until period end
The licence is still 'active' in D1. The app continues working normally. The customer gets everything they paid for until the renewal date.
During this period, POST /license/verify still returns {"valid": true, "status": "active"}. Nothing changes for the customer's experience until the period end date.
4
● Stripe
Period end arrives — Stripe deletes the subscription
At the exact moment current_period_end is reached, Stripe terminates the subscription and fires the deleted event.
{
  "type": "customer.subscription.deleted",
  "data": {
    "object": {
      "id": "sub_1ABC...",
      "status": "canceled",
      "ended_at": 1748649600
    }
  }
}
This event is the same one triggered by a final payment failure — the Worker handles it identically.
5
● D1 Database
Worker revokes the licence
The Worker updates the licence status to 'revoked' with a note. This happens automatically — no admin action needed.
UPDATE licenses
SET status = 'revoked',
    notes = 'Subscription cancelled by Stripe event'
WHERE stripe_subscription_id = 'sub_1ABC...'
6
● Customer App
Customer's next check-in shows access has ended
On the next app startup after the period end date, /license/verify returns {valid: false, status: 'revoked'}. Pro features are locked.
{"valid": false, "status": "revoked"}
The app should show a friendly message: "Your ModelCollectr Pro subscription has ended. Visit modelcollectr.co.uk to resubscribe." The customer's key won't work again unless you manually restore it in the admin dashboard (which would only make sense if they resubscribed through another route).
