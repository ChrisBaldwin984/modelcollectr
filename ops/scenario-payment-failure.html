Payment Failure & Recovery
A renewal charge fails — card expired, insufficient funds, or bank declined. The licence is suspended immediately. Stripe retries automatically over several days. This scenario has two possible outcomes: recovery or final revocation.
● Stripe ● Cloudflare Worker ● D1 Database ● Customer App
Step-by-Step Flow
11 steps with a decision fork. The outcome depends on whether Stripe's retries eventually succeed. Click any step for detail.
1
● Stripe
Stripe attempts renewal charge — card is declined
On the anniversary date, Stripe tries to charge the saved card. The bank returns a decline code (expired, insufficient funds, etc.).
Common decline reasons:
  • card_declined — bank refused (could be fraud, limit, etc.)
  • expired_card — card expiry date has passed
  • insufficient_funds — not enough balance
  • do_not_honor — generic bank refusal
Stripe's Smart Retries will try alternative processing routes first before generating the failure event.
2
● Stripe
Stripe fires invoice.payment_failed webhook
The failed invoice event is sent to the Worker immediately after the first decline.
{
  "type": "invoice.payment_failed",
  "data": {
    "object": {
      "subscription": "sub_1ABC...",
      "amount_due": 1999,
      "attempt_count": 1,
      "next_payment_attempt": 1716172800
    }
  }
}
next_payment_attempt is a Unix timestamp indicating when Stripe will try again. attempt_count tells you how many times it's tried so far.
3
● Worker
Worker verifies webhook and suspends the licence
The signature is verified, then the licence status is set to 'suspended' — not 'revoked'. This gives the customer time to fix their card before losing access.
UPDATE licenses
SET status = 'suspended'
WHERE stripe_subscription_id = 'sub_1ABC...'
Suspended vs Revoked:
  • Suspended — temporary, automatic. Stripe is still retrying. Can recover automatically.
  • Revoked — permanent (until admin restores). Stripe has given up or admin forced it.
4
● Customer App
Customer's next check-in returns {valid: false, status: 'suspended'}
On the next app startup, the licence check comes back as invalid. The app should show a "payment issue" message rather than a generic "invalid key" error.
POST /license/verify
{"key": "MC-K8HN-4XRQ-BZTW-7YJP"}

Response:
{"valid": false, "status": "suspended"}
The app should differentiate between status: "suspended" (payment problem — fixable) and status: "revoked" (permanent). A suspended message should tell the customer: "Your subscription payment failed. Please update your card at modelcollectr.co.uk/billing."
5
● Stripe
Stripe sends the customer a payment failed email
Stripe automatically emails the customer explaining the payment failed and providing a link to update their card. This is Stripe's own email — you don't need to do anything.
The email includes a secure link to a Stripe-hosted page where the customer can update their payment method. This is the primary recovery path. You can customise the email text in Stripe Dashboard → Settings → Emails → Failed payments.
6
● Stripe
Stripe retries the charge over the following days
Stripe's Smart Retry logic attempts the charge again at intervals — typically 3, 5, and 7 days after the first failure. The exact schedule is configurable in your Stripe settings.
Default Stripe retry schedule:
  • Attempt 1: Day 0 (immediate)
  • Attempt 2: Day 3
  • Attempt 3: Day 5
  • Attempt 4: Day 7 (final)
If the customer updates their card between attempts, Stripe will use the new card on the next retry. Configure in Stripe Dashboard → Settings → Billing → Retry schedule.
Decision Point — What happens next depends on whether a retry succeeds
✅ Outcome A — Recovery
  • Customer updates their card via Stripe's link
  • A retry succeeds → invoice.payment_succeeded fires
  • Worker sets status = 'active'
  • Customer's next check-in returns valid again
  • No new key — same key re-activates
❌ Outcome B — Final Failure
  • All retries fail (customer didn't update card)
  • Stripe cancels the subscription
  • customer.subscription.deleted fires
  • Worker sets status = 'revoked'
  • Customer permanently loses access
  • Must re-subscribe to get a new key
7
● Stripe — Outcome A: Recovery
A retry succeeds → invoice.payment_succeeded webhook fires
If the customer updated their card, or the bank approves on a retry, Stripe fires the success event.
{
  "type": "invoice.payment_succeeded",
  "data": { "object": { "subscription": "sub_1ABC..." } }
}
8
● D1 — Outcome A
Worker sets licence back to 'active'
The same UPDATE query as the renewal scenario. Status goes from 'suspended' → 'active'. The customer's next app check-in will succeed.
UPDATE licenses
SET status = 'active'
WHERE stripe_subscription_id = 'sub_1ABC...'
No admin action needed. The recovery is fully automatic.
9
● Stripe — Outcome B: Final Failure
All retries fail → Stripe cancels the subscription
After exhausting all retry attempts, Stripe marks the subscription as cancelled and fires customer.subscription.deleted.
{
  "type": "customer.subscription.deleted",
  "data": { "object": { "id": "sub_1ABC...", "status": "canceled" } }
}
Stripe also emails the customer to let them know their subscription has ended. The customer will need to re-subscribe and will receive a new licence key.
10
● D1 — Outcome B
Worker sets licence to 'revoked'
The subscription.deleted event causes the Worker to revoke the licence permanently. The customer loses Pro access on their next check-in.
UPDATE licenses
SET status = 'revoked',
    notes = 'Subscription cancelled by Stripe event'
WHERE stripe_subscription_id = 'sub_1ABC...'
At this point, the customer's key is dead. They must purchase a new subscription to get a new key. You do not need to do anything — this is fully automatic.
11
● Customer App — Outcome B
Customer's next check-in returns {valid: false, status: 'revoked'}
The app should show a clear message that their subscription has ended and direct them to the website to re-subscribe.
{"valid": false, "status": "revoked"}
The app message should be: "Your ModelCollectr Pro subscription has ended. Visit modelcollectr.co.uk to resubscribe and get a new licence key." The old key cannot be reactivated — a new subscription creates a new key.
What you might need to doIf a customer emails saying their app stopped working — check their subscription in Stripe. If it shows past_due or canceled you now know why. Direct them to update their card via the Stripe customer portal link, or if the subscription is already cancelled, tell them to re-subscribe.
