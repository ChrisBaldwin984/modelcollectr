Scenario 1 — Successful Purchase & Activation Happy Path

1
Customer visits modelcollectr.co.uk
Clicks "Get Pro — £19.99/yr" in the nav or hero, or the pricing card button.
2
Worker creates Stripe Checkout Session
POST /checkout → Stripe API → returns one-time checkout URL. 14-day free trial enabled. Promo code field shown on checkout page.
3
Customer completes Stripe checkout
Enters email + card details on Stripe-hosted page. Clicks Subscribe. Card is saved for annual auto-renewal.
4
Stripe fires customer.subscription.created webhook
Sent to https://api.modelcollectr.co.uk/webhooks/stripe. Verified by HMAC-SHA256 signature + 5-minute timestamp window.
5
Worker generates licence key & stores in database
Format: MC-XXXX-XXXX-XXXX-XXXX (cryptographically random). Stored in Cloudflare D1 with email, Stripe customer ID, subscription ID, status = active.
6
Licence email sent to customer
From hello@modelcollectr.co.uk via Resend. Contains key, activation steps, and subscription details.
7
Customer sees success page with key
Redirected to modelcollectr.co.uk/success?session_id=... — page polls /license/lookup every 2 seconds until key is ready (up to 30s). Key displayed with Copy button.
8
Customer activates key in desktop app
Help → Enter Licence Key → pastes MC-XXXX key → clicks Activate. App calls POST /license/verify → server confirms valid + active → Pro features unlock immediately.
9
App checks in on every launch
POST /license/verify on startup. Updates last_check_in timestamp in database. 7-day offline grace period — if server unreachable, keeps previous valid status for up to 7 days.

Scenario 2 — Annual Renewal Automatic

1
Stripe attempts annual charge
~12 months after subscription start (or after trial ends). Charges saved card automatically.
2
invoice.payment_succeeded webhook fires
Worker receives event → confirms subscription still active → no action needed. Licence key status remains active.
3
App continues working
Next check-in returns valid + active. Customer notices nothing — seamless renewal.

Scenario 3 — Payment Failure & Recovery Requires Action

1
Stripe annual charge fails
Card declined / expired. Stripe fires invoice.payment_failed webhook.
2
Licence suspended (not revoked)
Status set to suspended in database. App check-in returns valid: false, status: suspended. After 7-day grace period, Pro features lock in the app.
3
Stripe retries payment automatically
Stripe sends reminder emails to customer and retries the charge (typically 3–4 attempts over ~2 weeks depending on your Stripe retry settings).
Two outcomes
✓
Payment recovered: invoice.payment_succeeded fires → Worker sets status = active → App unlocks on next check-in. No intervention needed.
✗
Payment never recovered: Stripe cancels subscription → customer.subscription.deleted fires → Worker sets status = revoked → App locks permanently until new subscription.

Scenario 4 — Customer Cancels Subscription Graceful End

1
Customer cancels via Stripe customer portal
Or contacts you to cancel. Stripe cancels at end of current billing period (not immediately).
2
customer.subscription.deleted fires at period end
Worker receives event. Status set to revoked. Note: customer keeps access until their paid period expires — Stripe handles this timing.
3
App locks on next check-in after period ends
Pro features revert to free tier. Customer sees upgrade prompt. Their collection data is never deleted.

Scenario 5 — Admin Revocation (Refund / Abuse) Manual Action

1
Open admin dashboard
Visit https://api.modelcollectr.co.uk/admin with X-Admin-Secret header set in ModHeader browser extension.
2
Find the customer's row, click Revoke
Confirm the prompt. Status set to revoked immediately in database.
3
App locks on next check-in
Within 7 days at most (7-day grace period). Immediate if customer opens app while connected to internet.
4
Also cancel in Stripe if subscription is still active
Admin revoke only blocks the key — it doesn't stop Stripe charging the card. Go to Stripe Dashboard → Customers → cancel subscription separately.

Scenario 6 — Restore Revoked Licence Admin Action

1
Find the customer's row in admin dashboard
Status shows "revoked" (red badge).
2
Click Restore
Status immediately set to active in database.
3
App unlocks on next check-in
Customer opens app → check-in returns valid: true → Pro features unlock. No key change needed.

Scenario 7 — Customer Lost Their Key Email Support Action

1
Customer contacts you — can't find key email
Ask for the email address they used to subscribe.
2
Find their row in admin dashboard
Look up by email address in the table. Confirm status is active.
3
Click Resend
Same licence key email resent to their email address. No new key generated — same key as before.

Scenario 8 — Offline Grace Period Edge Case

1
App opens but can't reach the server
No internet, or server temporarily down. Check-in fails.
2
App checks last successful check-in date
If within 7 days: Pro features stay unlocked silently. Customer notices nothing.
After 7 days offline
→
Pro features lock. App shows message explaining it couldn't verify the licence and prompts to connect to internet.
→
Once connected, next check-in restores Pro immediately if key is still active.

Scenario 9 — Re-subscription (After Previous Cancellation) Edge Case

1
Customer subscribes again after previously cancelling
Goes through normal checkout flow again.
2
New key generated for new subscription
Old revoked key stays in database (not deleted). New key created for the new subscription ID. Customer gets a new email with a new key.
3
Customer enters new key in app
Help → Enter Licence Key → enters new MC-XXXX key → Pro unlocked again.
