Billing
Annual Renewal
Stripe automatically charges the customer's card on their anniversary date. If successful, Pro access continues without interruption and no action is needed from you or the customer.
● Stripe ● Cloudflare Worker ● D1 Database ● Customer App
Step-by-Step Flow
7 steps. This is entirely automated — no human action required unless something goes wrong. Click any step for detail.
1
● Stripe
Stripe schedules the renewal charge
Approximately 1 year after the original subscription was created (or after the 14-day trial ended), Stripe automatically generates a renewal invoice.
Stripe tracks current_period_end on each subscription. A few hours before this date, Stripe creates a new invoice for £19.99 and attempts to charge the saved payment method. You can see this in the Stripe dashboard under Subscriptions → the customer's subscription.
2
● Stripe
Stripe charges the customer's saved card
The payment is processed automatically. The customer sees a charge from "ModelCollectr" on their card statement.
Stripe uses the payment method that was saved during the original checkout. If the card has expired, Stripe may use a card updater (available for major UK banks) to try an updated number. A successful charge creates an Invoice with status paid.
3
● Stripe
Stripe fires invoice.payment_succeeded webhook
On successful payment, Stripe sends a webhook event to the Worker confirming the renewal invoice was paid.
POST /webhooks/stripe

{
  "type": "invoice.payment_succeeded",
  "data": {
    "object": {
      "subscription": "sub_1ABC...",
      "amount_paid": 1999,
      "currency": "gbp",
      "status": "paid"
    }
  }
}
This event is also fired for the initial (£0) trial invoice, but the Worker only acts on it when a subscription ID is present — which covers both cases safely.
4
● Worker
Worker verifies the webhook signature
As with all webhooks, the HMAC-SHA256 signature is verified and the 5-minute timestamp window is enforced before processing.
Standard signature verification applies (same as in the purchase flow). If verification fails, HTTP 400 is returned and Stripe will retry the webhook delivery.
5
● D1 Database
Worker ensures licence status is 'active' in D1
The licence row is updated to 'active' for this subscription. This is a no-op if it was already active, but recovers a 'suspended' licence if the customer had a previous payment hiccup.
UPDATE licenses
SET status = 'active'
WHERE stripe_subscription_id = 'sub_1ABC...'
This is safe to run even if the status is already 'active'. It also acts as the recovery step if the previous invoice had a brief failure (status was 'suspended') but then the customer updated their card before Stripe gave up.
6
● Stripe
Stripe sends the customer a renewal receipt email
Stripe automatically sends a payment receipt to the customer's email address. This is Stripe's own email — separate from anything sent by Resend.
You can customise Stripe's receipt emails in the Stripe Dashboard under Settings → Emails. The receipt shows the amount charged, the billing period, and a download link for the invoice PDF. No action needed from you.
7
● Customer App
Customer's app continues working without any action
On the next app startup, /license/verify returns {valid: true, status: 'active'} as normal. The customer experiences no interruption.
The licence key doesn't change on renewal — same key works indefinitely while the subscription is active. The customer doesn't need to re-enter their key or do anything. The only evidence the renewal happened is a £19.99 charge on their card statement.
