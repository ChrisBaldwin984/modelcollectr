Re-subscription After Cancellation
A lapsed customer decides to resubscribe. They go through the full purchase flow as if they were a new customer — a brand new licence key is generated and issued. Their old revoked key stays in D1 for records but is never reactivated.
● Customer ● Cloudflare Worker ● Stripe ● D1 Database ● Resend Email
Step-by-Step Flow
10 steps. Mostly the same as the purchase scenario, but with key differences around idempotency and Stripe's handling of returning customers. Click any step for detail.
1
● Customer
Lapsed customer revisits modelcollectr.co.uk and clicks Buy Now
They may be using the same email address as before, or a different one. Either way, the flow starts fresh from their browser.
There's no "returning customer" flow on the website — it's identical to a first-time purchase. The same startCheckout() call is made. The Worker doesn't look up old subscriptions — it just creates a new Stripe Checkout Session.
2
● Worker
Worker creates a new Stripe Checkout Session
Same as the original purchase — a new session is created with the annual price. No mention of the previous subscription.
The Checkout Session parameters are identical to the original purchase. If the returning customer uses the same email, Stripe will recognise them and may pre-fill their details.
3
● Customer
Customer completes the Stripe checkout
If Stripe recognises their email, they may see their previous payment method pre-filled. They complete the checkout and subscribe.
Stripe may link this to their existing Customer object (if the email matches) or create a new one. Either way, a new Subscription object is created with a new subscription ID. The 14-day free trial applies again as it's a new subscription.
4
● Stripe
Stripe creates a NEW subscription and fires customer.subscription.created
A brand new subscription ID is generated — different from the cancelled one. This is the key difference from the original purchase.
The webhook event contains: 
{
  "type": "customer.subscription.created",
  "data": {
    "object": {
      "id": "sub_2NEW...",     ← NEW subscription ID
      "customer": "cus_1XYZ...", ← may be same customer
      "status": "trialing"
    }
  }
}
The old subscription ID was sub_1ABC... (cancelled, row in D1 with status revoked). This new one is completely different.
5
● Worker
Worker verifies webhook and checks idempotency
The idempotency check looks for an existing key with THIS subscription ID — which doesn't exist yet. So the Worker proceeds to create a new key.
SELECT key FROM licenses
WHERE stripe_subscription_id = 'sub_2NEW...'

→ No row found → proceed to create new key
Critical: The idempotency check is on stripe_subscription_id, not on stripe_customer_id. This means the same customer can have multiple rows in D1 — one for each subscription. That's by design: each subscription period gets its own unique key.
6
● Stripe
Worker fetches customer email from Stripe
Same as original purchase — the email is fetched from the Stripe customer object and used to send the new key email.
If the customer used the same email, this is their existing Stripe Customer with the same ID. If they used a new email, Stripe may have created a new Customer object with a new ID.
7
● Worker
Worker generates a brand new licence key
A completely new MC-XXXX-XXXX-XXXX-XXXX key is generated. It has no relation to the previous key.
The old key (e.g. MC-K8HN-4XRQ-BZTW-7YJP) remains in D1 with status revoked. The new key (e.g. MC-R4MN-7ZPQ-XWTK-5HJC) is a completely different string generated fresh with crypto.getRandomValues().
8
● D1 Database
Worker inserts a NEW row in D1 for the new subscription
D1 now has two rows for this customer — the old revoked one and the new active one. The old row is preserved as a historical record.
-- Existing row (from original subscription):
key='MC-K8HN-...', stripe_subscription_id='sub_1ABC...', status='revoked'

-- New row (inserted now):
INSERT INTO licenses
  (key, stripe_customer_id, stripe_subscription_id, status, created_at)
VALUES
  ('MC-R4MN-7ZPQ-XWTK-5HJC',
   'cus_1XYZ...',   ← same customer ID
   'sub_2NEW...',   ← new subscription ID
   'active',
   1716500000000)
9
● Resend Email
Worker sends the new key via Resend
The customer receives a fresh licence email with the brand new key and activation instructions.
The email is identical to the original. The customer must use the new key — their old key is permanently revoked. They need to go into Help → Enter Licence Key and replace the old key with the new one.
10
● Customer
Customer enters the new key in the app — Pro restored
The customer removes or replaces the old key with the new one. /license/verify returns {valid: true, status: 'active'} and Pro features unlock.
The old key (MC-K8HN-...) will return {"valid": false, "status": "revoked"} if checked. The new key (MC-R4MN-...) returns {"valid": true, "status": "active"}. The customer must update the key stored in the app.
