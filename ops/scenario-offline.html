Edge Case
Offline Grace Period
Customer's PC has no internet connection — maybe they're on a flight, at a remote site, or their router is down. The desktop app must not immediately lock them out. A 7-day offline grace period is implemented in the app using locally cached check-in state.
● Customer App (local) ● Cloudflare Worker
Step-by-Step Flow
6 steps. This is entirely handled by the desktop app — the Worker and D1 are not involved during the offline period. Click any step for detail.
1
● Worker (last successful check-in)
App successfully verified the licence on last startup
The previous time the app ran with an internet connection, /license/verify returned {valid: true, status: 'active'}.
The app should store two things locally (e.g. Windows Registry or config file):
  • last_valid_check_timestamp — Unix timestamp of last successful verify
  • cached_status — "active" (only cache if valid=true)
These are written every time a successful check-in occurs, overwriting the previous values.
2
● Customer App
App starts up — no internet connection available
The app attempts to call /license/verify, but the network request fails (timeout, DNS failure, or connection refused).
The app should distinguish between:
  • A network error (can't reach the server) → use cached result
  • HTTP 200 with {"valid": false} → key is actually revoked — don't use cache
Network errors are a different category from a valid response saying the key is invalid.
3
● App Logic
App checks the cached check-in timestamp
The app reads the locally stored last_valid_check_timestamp and calculates how many days ago the last successful check-in was.
const lastCheck = readFromRegistry('last_valid_check_timestamp');
const daysSinceCheck = (Date.now() - lastCheck) / (1000 * 60 * 60 * 24);

if (daysSinceCheck <= 7 && cachedStatus === 'active') {
  // Grant access — within grace period
  unlockProFeatures();
  showOfflineWarning(7 - Math.floor(daysSinceCheck));
} else {
  // Grace period expired — lock features
  showGraceExpiredMessage();
}
4
● Customer
Pro features remain accessible during the grace period
The customer can use all Pro features normally for up to 7 days offline. A subtle banner reminds them to connect to the internet to re-verify.
Suggested UI message (non-intrusive): "Offline mode — Pro features active for X more days. Connect to the internet to verify your licence." This should be a persistent but non-blocking banner — not a modal that stops work.
5
● Customer reconnects → Worker
Internet restored — app checks in with the Worker
On next startup with internet, the app retries /license/verify. If the key is still active, the grace period timer resets.
A successful online check-in updates the stored timestamp to "now" — this resets the 7-day clock. The customer doesn't notice anything. The D1 check-in counter is also incremented on each successful verify.
6
● Customer App (grace expired)
If 7 days pass without a check-in — Pro is locked
After 7 days offline, the grace period expires. Pro features are locked. A clear message prompts the customer to connect to the internet to restore access.
Suggested message: "Your Pro licence requires internet verification every 7 days. Please connect to the internet and restart ModelCollectr to continue using Pro features. Your data and settings are safe."

As soon as they reconnect and restart, a successful /license/verify immediately unlocks Pro again — no action needed from you.
Notes on This Scenario
AspectDetail
Where is this logic?Entirely in the desktop app (Electron). The Worker and D1 are not involved during the offline period.
Grace period duration7 days from last successful check-in. Configurable in the app.
What's cached?Timestamp of last valid check-in + status value. Never cache an invalid status.
What about revocation?If the key is revoked while offline, the app won't know until it reconnects. After reconnecting, the next verify will return {valid: false} and the app should lock immediately (not use the cache).
App-side implementation requiredThis grace period is a desktop app feature — it is not enforced by the Worker. The Worker always returns the current live status. The app must implement the caching and grace period logic.
